# 🤖 РАСШИРЕННЫЙ ПАСПОРТ ПРОЕКТА "AiSiteFiller.V2" (ДЛЯ ИИ-НАПАРНИКА)
## 1. СТРОГИЕ ПРАВИЛА ВЗАИМОДЕЙСТВИЯ С РАЗРАБОТЧИКОМ
1.1. Категорический запрет на галлюцинации и домыслы: Тебе запрещено угадывать, предполагать или придумывать структуру методов, классов, таблиц СУБД или логику очередей, которых разработчик не прислал в текущем чате. Если точный код файла (например, MainForm.cs или фонового воркера) не загружен в сессию — остановись и потребуй текст файла у пользователя.
1.2. Обязательное правило линейки кода: Перед каждым блоком кода в специальном markdown-редакторе ты ОБЯЗАН писать строчку: "Код ниже — [X] строк", где вместо [X] указываешь точное число строк в выводимом блоке. Это жесткая защита от обрывов и багов визуального редактора платформы.
1.3. Принцип модульности ответов: Если выводимый код превышает 50 строк, ты обязан разбивать его на логические части. Пользователь сам склеит у себя. Никогда не пытайся выдать огромный монолит за один раз.
1.4. Прозрачность исключений и логирование: Категорически запрещено оставлять пустые блоки "catch {}" или проглатывать ошибки. Любой сетевой сбой, ошибка парсинга или падение API должны быть записаны в лог через интерфейс ILogger и проброшены выше по цепочке ("throw") для корректного срабатывания Retry-политики.
1.5. Режим работы по плану: Разработка ведется строго по пунктам интерактивного плана из файла TODO.md. Код пишется только тогда, когда разработчик дает прямую команду (например: "Реализуем пункт 2.2.1"). До этого момента ведется только концептуальное проектирование.
## 2. ТЕХНИЧЕСКИЙ ПАСПОРТ СИСТЕМЫ (АРХИТЕКТУРА И СТЭК)
2.1. Название решения (Solution): Решение создано в Visual Studio и называется строго "AiSiteFiller.V2".
2.2. Платформа и язык: C# (.NET Core / .NET 8 / .NET 9), тип приложения — Windows Forms (WinForms). Запуск локальный на ПК автора.
2.3. Слои Чистой Архитектуры (Clean Architecture):
* Слой "Domain" — чистые сущности бизнеса, модели данных статей, логов и очередей. Слой полностью изолирован от внешних библиотек.
* Слой "Application" — абстрактные интерфейсы сервисов (контракты), Use Cases координации. Здесь лежит единый интерфейс IPublisherService для веера.
* Слой "Infrastructure" — конкретные реализации: работа с PostgreSQL, MongoDB, HttpClient-ы для DeepSeek, WordPress REST API, Stable Diffusion и Ollama.
* Слой "UI (WinForms)" — визуальная морда приложения, конфигурация DI-контейнера (пула зависимостей) и статус-бар.
2.4. Реляционная база данных: PostgreSQL (локальная служба Windows). Отвечает за транзакционное хранение очередей, логов заголовков и Base64-текстов.
2.5. NoSQL база данных: MongoDB (локальная служба Windows). Отвечает за хранение бинарных данных картинок (GridFS) и хэшей промптов для защиты от дублей.
2.6. Конфигурация: Базовые настройки хранятся в "appsettings.json" (улетает в Git). Секретные токены, пароли баз и API лежат в "appsettings.local.json" (в .gitignore). Текстовые инструкции для ИИ лежат в отдельном файле "prompts.json" для изменения логики контента без пересборки .exe.
## 3. ТЕХНИЧЕСКИЕ ОГРАНИЧЕНИЯ И ЗАЩИТА ОТ БАГОВ (ИЗ ВЕРСИИ V2)
3.1. Правило сборки URL-строк: Из-за парсеров любые URL в C# собираются атомарной конкатенацией (`"https://" + "dzen." + "ru"`).
3.2. Правило вывода ссылок: В чате ссылки разделяются символами `+` для корректного отображения.
3.3. Кодирование кириллицы: Обязательное использование `Uri.EscapeDataString()` для слагов и параметров.
3.4. Изоляция Beget: PHP-скрипты на хостинге не хранятся в Git. При изменениях ИИ обязан запрашивать их структуру у пользователя.
3.5. Маскировка ссылок: Допускается запрос структуры `.htaccess` с редиректом `^go/(.*)$` для CPA-партнерок.

## 4. БИЗНЕС-ЛОГИКА И МОНЕТИЗАЦИИ
4.1. Цель: Автономный пассивный доход через сетку сайтов (РСЯ, CPA Яндекс.Маркета).
4.2. Трафик: Бесплатный органический трафик за счет высокого траста площадок ("запостил и забыл").
4.3. Защита от фильтра МПК: ИИ выступает как аналитик, обрабатывая 30-40 отзывов для создания уникального контента, а не просто "воды".
4.4. Кириллические слаги: При формировании адресов страниц на WordPress обязательна предварительная обработка через `Uri.EscapeDataString()`, иначе визуальный редактор CMS ломает вложенные теги разметки.
4.5. Инфраструктура хостинга Beget: На серверах beget.com размещаются веб-сайты и закрытые легковесные PHP-скрипты, принимающие POST-запросы и изображения от приложения. Эти PHP-скрипты являются частью закрытой инфраструктуры и категорически запрещены к добавлению в публичный Git-репозиторий. Любые изменения в механизме сетевого обмена требуют от ИИ предварительного запроса у пользователя актуальной структуры скриптов хостинга, чтобы исключить поломку интеграции. Все ручные правки на сторонних сервисах фиксируются внутри репозитория, обеспечивая доступ к истории конфигураций из одной точки.
4.6. Разделение файлов конфигураций: Настройки приложения строго делятся на три независимых файла. Базовый `appsettings.json` содержит несекретную структуру и улетает в Git. Файл `appsettings.local.json` содержит приватные токены, пароли баз данных и ключи приложений — он заносится в `.gitignore` и никогда не публикуется. Файл `prompts.json` хранит все системные инструкции для ИИ, Stable Diffusion и Ollama, позволяя точечно менять логику контента и стилистику изображений без необходимости пересборки исполняемого `.exe` файла.

## 5. ЗАДЕЛ ПОД ПАРАЛЛЕЛЬНЫЙ БУРЖ (INTERNATIONAL EXTENSIBILITY)
5.1. Мультиязычный контекст: Архитектура приложения изначально должна проектироваться с учетом параллельного масштабирования на международный US/EU сегмент. Все модули, промпты и очереди обязаны поддерживать переключение языкового контекста (флаги локализации `RU` / `EN`).
5.2. Характер контента под регионы: Система должна разделять типы генерации. Для RU-сегмента генерируется преимущественно информационный контент, оптимизированный под рекламные баннеры РСЯ. Для Бурж-сегмента конвейер должен переключаться на генерацию продающих обзоров и сравнительных тест-драйвов, заточенных под западные CPA-партнерки.
5.3. Безопасность CPA-маркеров: При интеграции реферальных ссылок внутри текстовых массивов используются специализированные маркеры-плейсхолдеры с разделителями `|||`. Данный формат минимизирует риски ложного срабатывания регулярных выражений C# при обработке «грязных» текстовых строк.
5.4. Разделение потоков планировщика: Фоновые воркеры очереди должны поддерживать параллельное независимое планирование. "Поток RU" распределяет анонсы на WordPress, Дзен и Пикабу, в то время как будущий "Поток US" должен без изменения ядра системы отправлять данные на Medium, Substack и аналогичные западные платформы.
5.5. Чистота абстракций издателей: Интерфейсы публикации в слое `Application` обязаны оставаться универсальными. Добавление условного `MediumPublisherService` должно происходить путем создания нового класса в слое `Infrastructure` по аналогии с существующими, не затрагивая внутренние слои Чистой Архитектуры.
5.6. Кэширование с флагом локализации: Документы в MongoDB и строки в PostgreSQL обязаны содержать обязательное поле региона или языка (`Language`), чтобы базы данных корректно разделяли и изолировали кэшированные статьи на русском и английском языках, исключая смешивание потоков трафика.

## 6. ПРОЕКТИРОВАНИЕ ХРАНИЛИЩ ДАННЫХ (БАЗЫ ДАННЫХ)
6.1. Реляционный слой (PostgreSQL): Отвечает за транзакции и логику очередей. Содержит три ключевые таблицы. Таблица `articles` хранит метаданные, названия продуктов, языковые флаги, исходный JSON от ИИ и Base64-блоки текстов. Таблица `publishing_queue` выполняет роль планировщика (ID задачи, ID статьи, целевая платформа, статус `Pending/Processing/Posted/Error`, счетчик попыток `retry_count`, текст ошибки `last_error` и время следующего запуска `scheduled_at`). Таблица `used_titles_log` сохраняет историю опубликованных заголовков по платформам с меткой времени.
6.2. Слоистый NoSQL контур (MongoDB): Изолирован от текстовой базы. Отвечает за хранение тяжелых бинарных данных изображений (GridFS/Блобы) и историю уникальных графических промптов. Перед запуском Stable Diffusion C#-код проверяет наличие хэша текущего промпта в MongoDB: если совпадение найдено — программа мгновенно забирает готовую картинку из Mongo, полностью освобождая видеокарту от повторного рендеринга.

## 7. АЛГОРИТМ РЕЗИЛЕНТНОСТИ (RETRY PATTERN) И ДИАГНОСТИКА
7.1. Интеграция библиотеки Polly: Все внешние сетевые операции (запросы к ИИ, вызовы WP REST API, отправка анонсов на веер, обращение к локальным SD и Ollama) защищены политикой Polly. Используется стратегия 3 повторных попыток с экспоненциальной задержкой и добавлением случайного джиттера для маскировки под человека.
7.2. Прозрачность исключений: Блоки `catch` используются строго для детального логирования стека через `ILogger` с последующим обязательным пробросом ошибки (`throw`) выше по цепочке. Проглатывание ошибок и сокрытие исключений запрещены.
7.3. Логика экстренной остановки: Если Polly исчерпывает все 3 попытки, задача в Postgres получает статус `Error`, конвейер для данной статьи приостанавливается, а независимая служба `TelegramAlertService` отправляет технический лог с описанием сбоя в ваш админ-чат.
7.4. Фоновый статус-бар здоровья системы: В слой `UI` (Windows Forms) интегрирована нижняя панель индикаторов. При каждом старте приложение в фоновых асинхронных потоках опрашивает официальные бесплатные API Яндекс.Метрики (счетчик визитов), Яндекс.Вебмастера (контроль критических фильтров МПК/банов поисковиков) и API хостинга Beget (остаток баланса счета и свободного места на диске). В случае падения метрик или критического расхода диска статус-бар мгновенно меняет цвет, сигнализируя о необходимости ручного вмешательства.
7.5. Поштучный выкат с паузами: Локальный воркер обрабатывает очередь строго последовательно. Программа публикует не более 1 полной статьи на сайт и не более 1 анонса на внешние платформы в сутки. Между публикациями закладываются случайные временные интервалы, что полностью имитирует поведение живого автора и защищает новые аккаунты от спам-фильтров.
